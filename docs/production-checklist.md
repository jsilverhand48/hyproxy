# Production checklist

Standing hyproxy up in production, from a bare host to a hardened, exposed
service. `install.sh` automates the mechanical parts; this checklist is the
authoritative list of everything, including what it cannot do for you. Supported
platform: Rocky Linux only. Read with `docs/production.md` (the hardening
rationale) and `docs/deployment.md` (topology).

Exposure gate: `install.sh` brings the stack fully up (firewall open, ingress
running), but nobody can reach it until you add public DNS records (a manual
step). Treat "add public DNS" as the last action, taken only after the hardening
items below are done.

## 1. Prerequisites (before running install.sh)

- [ ] A Rocky Linux host with a public IP and inbound `:443` reachable.
- [ ] A domain you control, hosted on a DNS provider that lego supports
      (https://go-acme.github.io/lego/dns/) with API access enabled.
- [ ] That provider's API credentials to hand (key/secret or token).
- [ ] Decided the install directory (default `/opt/hyproxy`).

## 2. One-command install

```sh
curl -fsSL https://raw.githubusercontent.com/YOUR_ORG/YOUR_REPO/main/install.sh | sh
```

Prefer to review first (it prompts for secrets and runs as root):

```sh
curl -fsSL https://raw.githubusercontent.com/YOUR_ORG/YOUR_REPO/main/install.sh -o install.sh
less install.sh && sh install.sh
```

It prompts for the domain, admin identity, DNS-01 provider credentials, Postgres
password (auto-generated by default), and whether to enable guac.

### What install.sh does for you (verify after)

- [ ] Host dependencies installed (Docker + compose, Go, make, uv, lego).
- [ ] Firewall: `:443` opened in firewalld; control-plane ports stay loopback.
- [ ] `.env` and root-owned `/etc/hyproxy/acme.env` written.
- [ ] Container images built; DB migrated; signing keys generated; first admin
      and the `admin-ui` OIDC client created (save the one-time password it
      prints).
- [ ] Let's Encrypt wildcard cert issued (staging dry-run first, then real) and
      installed; `hyproxy-acme.timer` enabled for renewal.
- [ ] Data plane deployed under systemd (`hyproxy-dataplane`, de-privileged
      `hyproxy` user, SELinux `bin_t`) and control plane containers up.

Quick verification:

```sh
cd /opt/hyproxy && docker compose --profile app ps          # all healthy
systemctl status hyproxy-dataplane --no-pager               # active (running)
systemctl list-timers hyproxy-acme.timer                    # scheduled
```

## 3. Still manual (do these before adding public DNS)

### Networking
- [ ] Public DNS `A`/`AAAA` records for `idp.`, `admin.`, `auth.`, and each app
      host, pointing at the server IP. (This is the exposure gate; do it last.)
- [ ] If the public IP is dynamic, DDNS: the decision core is `ops/ddns.py`, but
      you must implement a `DnsProvider` and run `update_if_needed` on a timer.

### Admin access (out of band)
- [ ] WireGuard for the admin plane (`docs/admin-access.md`). In production the
      admin API is not exposed to the internet or LAN; reach it over WireGuard
      only. Adjust `dataplane/config.json` so the admin host is not fronted
      publicly, then restart the data plane.

### Secrets backend (the one real code task)
- [ ] Migrate off the file backend to TPM (`docs/production.md` section 1):
      implement `core/secrets.tpm_unseal` (currently raises `NotImplementedError`),
      seal a master key to the TPM, add it and `rotate-master-key`, switch
      `HYPROXY_SECRETS_BACKEND=tpm`, then destroy the on-disk key. The file
      backend that install.sh configures keeps the master key on disk and is a
      bridge only.

### Backend and backchannel TLS
- [ ] Enforce backend TLS verification (no skip-verify); trust or pin an internal
      CA before enabling any https backend.
- [ ] Retire the dev-only `idp_verify_tls=false` on the authz to IdP backchannel.

### Network segmentation
- [ ] Confirm the control-plane containers publish on `127.0.0.1` only, guacd
      stays internal, and only `:443` plus the WireGuard admin path face any
      network.

### Observability
- [ ] Off-box logging: cron `docker compose run --rm cli ship-logs` into an
      append-only collector the host cannot delete from; alert on
      `severity:"high"` (especially `login.break_glass.used`,
      `oidc.refresh.reuse_detected`).
- [ ] NTP/clock sync (TOTP and token expiry depend on it).

### Quality and security gates
- [ ] Run the gates that install.sh skips by default: `make audit` (bandit,
      pip-audit), `make dp-test` (gofmt, vet, tests), `make check`. (Re-run
      install with `HYPROXY_RUN_GATES=1` to include them in bootstrap.)
- [ ] The dedicated security review against `docs/security-notes.md`; resolve
      every dev-only accepted risk.

## 4. First login and app onboarding

- [ ] Sign in at `https://idp.<domain>/auth/login` as the admin using the
      one-time password from the install output; enroll two passkeys at
      `/auth/enroll/webauthn` (admin tier requires two).
- [ ] Register each protected app as a relying party
      (`docker compose run --rm cli create-client ...`), add its route to
      `dataplane/config.json`, and restart the data plane (routes are not
      hot-reloaded; see `UPDATES.md`).

## 5. Go-live verification

```sh
curl --resolve idp.<domain>:443:127.0.0.1 https://idp.<domain>/.well-known/openid-configuration
curl --resolve nope.<domain>:443:127.0.0.1 -o /dev/null -w '%{http_code}\n' https://nope.<domain>/   # 421
```

Green: TLS verifies against real Let's Encrypt, idp discovery returns the
expected `issuer`, unknown Host returns 421, OIDC login (PKCE + DPoP) completes,
and a mutation triggers the WebAuthn step-up.

## 6. Go-live gate

Only after the section 3 items are done and the security review is signed off,
add the public DNS records. That is the moment the service becomes reachable.
Ongoing changes and redeploys: `UPDATES.md`.
